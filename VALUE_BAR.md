# Value Bar

The repo standard is simple: nothing is treated as valuable until it is adversarially-confirmed to add value.

Mechanical checks are necessary. They are not enough. A passing gate proves the code and docs are internally consistent. It does not prove the tool helped a founder, operator, or builder do better work.

## What Counts

A module, skill, demo, or claim can be called valuable only when all of these are true:

1. A skeptical user or reviewer tried it on a real or realistic workflow.
2. The workflow had a baseline, such as the old process, a manual pass, or a competing tool.
3. The outcome improved on a named axis, such as time saved, clarity, quality, cost, activation, risk caught, or decision speed.
4. The reviewer had room to reject it, and their objections were recorded.
5. The evidence is saved in a receipt that another person can inspect.

## Labels

Use these labels instead of vague confidence words:

- `candidate`: runnable, internally tested, not yet proven useful.
- `mechanically vetted`: tests, gates, receipts, and docs agree.
- `adversarially confirmed`: a skeptical review or user run found clear value against a baseline.
- `rejected`: the tool ran, but did not add value or made the workflow worse.

Do not use "vetted", "validated", "proven", or "adds value" without the evidence that earns the label.

## Receipts

Every value receipt should name:

- the user or reviewer role, without private personal details
- the task and baseline
- the artifact or command tested
- the objection or counterexample tried
- the measured or observed improvement
- the follow-up change, if the tool failed

If the evidence is missing, say that directly. The honest status is better than a polished claim.
